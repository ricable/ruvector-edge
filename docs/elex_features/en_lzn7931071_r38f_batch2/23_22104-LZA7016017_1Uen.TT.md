# 1 NR Mobility Overview

NR Mobility provides the fundamental mobility functions to UEs in the

network.

| Feature Name           | NR Mobility                                             |
|------------------------|---------------------------------------------------------|
| Feature Identity       | FAJ 121 5041                                            |
| Value Package Name     | NR Base Package                                         |
| Value Package Identity | FAJ 801 4002                                            |
| Node Type              | Baseband Radio Node                                     |
| Access Type            | NR                                                      |
| Licensing              | Non-license-controlled feature. No license is required. |

Summary

NR Mobility, providing basic mobility functions, enables gNodeBs to perform

measurement-based handover, PSCell change, and redirection of UEs in the network. In

addition, the feature provides idle mode mobility support to the UEs.

Additional Information

For more information, see the following documentation:

- 3GPP TS 38.300; NR; Overall Description; Stage-2
- 3GPP TS 38.331; NR; Radio Resource Control (RRC); Protocol Specification

## 3.1 Mobility Procedures for EN-DC

### 3.1.1 Evaluation of PCI for PSCell Change

PSCell change is based on PCI reporting from

measurement reports.

When a UE measurement report is received, the gNodeB searches for a target cell MO with

the reported PCI. The target cell is represented by the following MO classes:

- Cells on the same gNodeB are represented by an instance of the NRCellCU MO.
- Cells on a neighbor gNodeB are represented by an instance of the ExternalNRCellCU MO.

Before a mobility action is triggered, the gNodeB checks the reported PCI based on

the following criteria:

- The target cell has a reference to the NR frequency reported by the UE. The following attributes are used: Table 3 NR Frequency References Attributes Cell Location NRCellCU.nRFrequencyRef On the same gNodeB ExternalNRCellCU.nRFrequencyRef On a neighbor gNodeB
- The source cell has a cell relation with the target cell. The following attributes are used: Table 4 Cell Relations between the Source and the Target Cell Reference from the Source Cell Representation of the Target Cell Cell Location NRCellRelation.nRCellRef NRCellCU On the same gNodeB ExternalNRCellCU On a neighbor gNodeB
- The NRCellRelation.isHoAllowed attribute is set to true for the target cell.
- The target cell supports at least one PLMN that is valid for mobility. The following alternatives exist: If an HRL is not sent for the UE from the core network, the selected PLMN must be supported in the target cell. The supported PLMNs for a cell are listed in the NRCellCU.pLMNIdList attribute. If an HRL is sent for the UE from the core network, at least one PLMN must be allowed by the HRL. The following criteria apply to the PLMN: The PLMN is supported in the target cell. NR RAN is not restricted for the PLMN. The TAC of the target cell is allowed and not forbidden for the PLMN by the HRL.
- In case of intra-gNodeB PSCell change, the UeMCCellProfile.connModeUeAllowedToBarredCell attribute must be set to true in the target cell if the NRCellDU.cellBarred attribute is set to BARRED in the target cell. If the target cell is barred and the UeMCCellProfile.connModeUeAllowedToBarredCell attribute is to false, the target cell is considered to be an invalid PSCell change target.
- In case of intra-gNodeB PSCell change, the UeMCCellProfileUeCfg.connModeUeAllowedToResCell attribute must be set to true in the target cell if the AdditionalPLMNInfo.cellReservedForOperator or the NRCellDU.cellReservedForOperator attribute is set to RESERVED in the target cell. If the target cell is reserved and the UeMCCellProfileUeCfg.connModeUeAllowedToResCell attribute is set to false, the target cell is considered to be an invalid PSCell change target. Note: The UeMCCellProfileUeCfg.connModeUeAllowedToResCell attribute can be configured for different UE groups.
- During inter-gNodeB PSCell change, forbidden TAI is considered.

If all requirements are fulfilled, the PCI is a valid target for mobility.

### 3.1.2 Intra-gNodeB PSCell Change

PSCell change can be initiated for the UE if PCI evaluation and UE capabilities allow

PSCell change to the target cell.

Figure 1   Intra-gNodeB PSCell Change

If the target neighbor cell fulfills all the criteria on the presence of PLMN and TAI

identities, the PSCell change is performed using the following signaling sequence:

1. The source Secondary Node sends the X2AP SGNB MODIFICATION REQUIRED message to the Master Node. The message contains the target neighbor cell configuration. The NR user data traffic is suspended in the current serving PSCell.
2. After the reception of the X2AP SGNB MODIFICATION REQUIRED message, the Master Node sends the RRCConnectionReconfiguration message to the UE. The message contains the target PSCell configuration.
3. When reconfigured, the UE sends the RRCConnectionReconfigurationComplete message to the eNodeB.
4. At reception of the RRCConnectionReconfigurationComplete message, the eNodeB sends the X2AP SGNB MODIFICATION CONFIRM message to the gNodeB. Intra-node packet forwarding is performed for the modified DRBs in the gNodeB.
5. The UE performs the NR Random Access Procedure to the new serving PSCell. The NR user data traffic is resumed in the new serving PSCell.

#### 3.1.3.1 Inter-gNodeB PSCell Change without LTE Intra-Cell Handover

If the number of the DRBs is less than or equal to eight, inter-gNodeB PSCell change

can be performed without LTE intra-cell handover.

This method provides uninterrupted user plane connection in LTE during the PSCell

change.

Figure 2   Inter-gNodeB PSCell Change without LTE Intra-Cell Handover

If the target neighbor cell fulfills all the criteria on the presence of PLMN and TAI

identities, the PSCell change is performed using the following signaling sequence:

1. The source Secondary Node sends the X2AP SGNB CHANGE REQUIRED message to the eNodeB triggering the PSCell change. The message contains the PCI of the target neighbor cell, and the identity of the target Secondary Node hosting the target neighbor cell. During inter-gNodeB PSCell change to an NR cell with MORAN configuration, the Master Node with the Automated Neighbor Relations feature activated can find the target Secondary Node hosting the target cell, if one of the following conditions are met, in this order: The global gNodeB ID in the Target SgNB ID IE sent in the X2AP SGNB CHANGE REQUIRED message is matched with the global gNodeB ID of the ExternalGNodeBFunction MO instance representing the target Secondary Node in the Master Node. If the global gNodeB ID does not match, but the following parameters are matched: The gNodeB ID in the Target SgNB ID IE sent in the X2AP SGNB CHANGE REQUIRED message with the ExternalGNodeBFunction.gNodeBId attribute. The PLMN in the Target SgNB ID IE with the first PLMN of the ExternalGUtranCell.plmnIdList attribute. The ExternalGUtranCell MO instance represents the target cell with MORAN configuration on the Master Node.
2. At the reception of the X2AP SGNB CHANGE REQUIRED message, the Master Node applies the following procedure: The Master Node evaluates the LTE-to-NR cell relation between the serving LTE cell and the target NR cell. If the cell relation exists and the Master Node is configured to restrict EN-DC setup for the NR cell, the PSCell Change is rejected. The Master Node verifies if the target PSCell contains the following information: At least one of the Serving or Equivalent PLMN identities listed in the HRL match one of the identities configured in the LTE PCell. At least one TAI of the target neighbor cell is not forbidden. The Master Node checks if the target frequency is blocklisted. If the target frequency is blocklisted, the Master Node sends the X2AP SGNB CHANGE REFUSE message to the source Secondary Node. If the target PSCell fulfills all the criteria on the presence of PLMN and TAI identities, the Master Node sends the X2AP SGNB ADDITION REQUEST message to the target Secondary Node. The message contains configuration information on the target PSCell.
3. The target Secondary Node sends the X2AP SGNB ADDITION REQUEST ACKNOWLEDGE to the Master Node. The message contains information on TEID to be used for packet forwarding.
4. The Master Node sends the following messages in parallel: The X2AP SGNB CHANGE CONFIRM message to the source Secondary Node. The message contains the TEID received from the target gNodeB. At the reception of the X2AP SGNB CHANGE CONFIRM message, the following sequence applies: The NR user data traffic is suspended in the source PSCell. The source Secondary Node starts the packet forwarding to the target Secondary Node in downlink. After the packet forwarding is started, the source Secondary Node sends the X2AP SN STATUS TRANSFER message to the Master Node. If configured, the source Secondary Node sends the X2AP SECONDARY RAT DATA USAGE REPORT message to the Master Node. The Master Node sends the X2AP SN STATUS TRANSFER message to the target Secondary Node with the COUNT value for packet forwarding. The Master Node sends the RRCConnectionReconfiguration message to the UE.
5. When reconfigured, the UE sends the RRCConnectionReconfigurationComplete message to the Master Node.
6. At the reception of the RRCConnectionReconfigurationComplete message, the Master Node sends the X2AP SGNB RECONFIGURATION COMPLETE to the target Secondary Node.
7. The Master Node sends the S1AP E-RAB MODIFICATION INDICATION message to the MME. ### 3.2.2 Handover Type Selection

During NR intra-system handover, the source gNodeB can initiate an NG or an Xn-based

handover. During inter-system handover form NR to LTE, the gNodeB initiates an NG-based

handover.

Figure 4   Handover Type Selection

The gNodeB selects the handover type based on the following process:

1. The source gNodeB checks if the handover target is the same RAT. If the target cell is on a neighbor eNodeB, an IRAT handover to LTE is initiated using NGAP signaling.
2. If the target is the same RAT, the source gNodeB checks if the target cell is within the area of the source gNodeB or within the area of a neighboring gNodeB. If the target cell is within the area of the source gNodeB, intra-gNodeB handover is initiated. Note: If the serving PLMN changes during the procedure, NG-based handover is initiated.
3. If the target cell is within the area of a neighboring gNodeB, the value of the GNBCUCPFunction.prefInterGnbHo attribute is evaluated. If it is set to NG\_HO, the source gNodeB triggers an NG-based handover.
4. If the GNBCUCPFunction.prefInterGnbHo attribute is set to XN\_HO, the source gNodeB checks if the target gNodeB is in the same AMF region as the source gNodeB. If the source gNodeB is not in the same AMF region as the target gNodeB, the source gNodeB triggers an NG-based handover.
5. If the source gNodeB is in the same AMF region as the target node, the source gNodeB checks if there is an available Xn connection between the source gNodeB and the target gNodeB. If there is no Xn connection between the source gNodeB and target gNodeB, the source gNodeB triggers an NG-based handover.
6. If an Xn connection exists between the source and target gNodeBs, the source gNodeB checks if the UE is configured with NR-DC. Based on the result of the check, the following handover options are applicable: If the UE is configured with NR-DC, the source gNodeB triggers an NG-based handover. If the UE is not configured with NR-DC, the source gNodeB performs an Xn-based handover.

#### 3.2.2.4 IRAT Handover to LTE

IRAT handover in NR SA to LTE can be initiated if the PCI evaluation and the UE

capabilities allow IRAT handover to the target LTE cell.

The purpose of an IRAT handover is to move the UE in RRC\_CONNECTED mode

to a cell using another RAT.

IRAT handover from NR SA to LTE is executed according to the following sequence:

Figure 8   Sequence of IRAT Handover to LTE

1. The source gNodeB receives a B2 measurement report from the UE in the RRC MeasurementReport message containing a PCI of a target LTE frequency.
2. If the PCI evaluation is successful, the source gNodeB sends the NGAP HANDOVER REQUIRED message to the AMF. The source gNodeB includes the E-RAB ID information and the downlink data forwarding information proposed for each E-RAB ID received from the 5GC.

The MME receives the direct packet forwarding information from the AMF. Handover resource allocation is done in the target eNodeB.
4. The target eNodeB replies with the S1AP HANDOVER REQUEST ACKNOWLEDGE message to the MME. The confirmed direct packet forwarding information is forwarded to the AMF through the MME.
5. The AMF sends the NGAP HANDOVER COMMAND message to the source gNodeB to execute the handover to the target LTE cell. 7. The UE confirms the reconfiguration by sending the RRC RRCReconfigurationComplete message to the target eNodeB.
8. The target eNodeB sends the S1AP HANDOVER NOTIFY message to the MME with information on handover execution.
9. The AMF instructs the source gNodeB to release the UE context for the UE transferred to LTE by sending the NGAP UE CONTEXT RELEASE COMMAND message.
10. At UE release, the source gNodeB sends the NGAP UE CONTEXT RELEASE COMPLETE message to the AMF.

support of direct packet forwarding during IRAT inter-system handover. As a result, IRAT

#### 3.3.2.4 Search Zone for Mobility Control at Poor Coverage for EN-DC

UEs in the search zone search for cells on other frequencies with better

coverage.

When the UE connects to the cell, and there are NR inter-frequency candidates for which

the UE can perform measurements, the UE is configured with the A2 Entering Search Zone

measurement to detect when the UE enters the search zone. This measurement is configured

to use the trigger quantity specified in the

McpcPSCellProfileUeCfg.mcpcQuantityList attribute, with separate

measurements being configured if more than one trigger quantity is specified. When the

UE is in the search zone, it is in poor coverage.

When the UE enters the search zone, it is reconfigured with the following

measurements:

- A set of A5 NR Inter-Frequency Candidate measurements to search for NR inter-frequency PSCell candidates
- The A1 Leaving Search Zone measurement to detect when the UE moves into good coverage
- The A2 Critical Coverage measurement to detect when the UE moves into critical coverage

Each measurement uses the same trigger quantity as that of the received A2

Entering Search Zone measurement.

When these measurements are configured for the UE, the A2 Entering Search Zone

measurement is removed. Only the A2 Entering Search Zone measurement with the trigger

quantity corresponding to the received measurement is removed, and another A2 Entering

Search Zone measurement might still be configured in the UE for another trigger

quantity.

A timer can be used to limit how long the UE can search for cells on other NR

frequencies. This reduces the negative impact that measurements have on throughput and

UE battery consumption. The timer can be enabled separately for each trigger quantity by

setting one or more of the following attributes to a value other than

-1:

- McpcPSCellProfileUeCfg.rsrpSearchTimeRestriction
- McpcPSCellProfileUeCfg.rsrqSearchTimeRestriction
- McpcPSCellProfileUeCfg.sinrSearchTimeRestriction

If there are no NR inter-frequency candidates for which the UE can perform measurements,

the A2 Entering Search Zone measurement is not configured. However, if no such NR

inter-frequency candidates exist and critical coverage measurement is enabled in the

current PSCell, the UE is configured with the A2 Critical Coverage measurement. The

critical coverage measurement can be enabled for each trigger quantity separately by

setting one or more of the following attributes to true:

- McpcPSCellProfileUeCfg.rsrpCriticalEnabled
- McpcPSCellProfileUeCfg.rsrqCriticalEnabled
- McpcPSCellProfileUeCfg.sinrCriticalEnabled

#### 3.3.2.5 Evaluation of Frequency Relations for EN-DC

The feature relies on a set of frequency relations that is used to decide where the

UE can make measurements to search for PSCell candidates on other NR

frequencies.

The NR frequency selection is based on the

following criteria:

- If the frequency relation only allows certain PLMNs and the UE does not have an HRL, then at least one PLMN must be the serving PLMN.
- If the frequency relation only allows certain PLMNs and the UE has an HRL, then at least one PLMN must be common between the PLMN list and the HRL. Note: PLMN lists are specified on the frequency relation level and on the cell level. If these lists are set inconsistently, the UE can make measurements on too many or too few frequencies.
- The UE must be capable of measurement on the target frequency.
- The McpcPSCellNrFreqRelProfileUeCfg.inhibitMeasForCellCandidate attribute must be set to false for the target frequency.
- The UeMCNrFreqRelProfileUeCfg.connModeAllowedPSCell attribute must be set to true for the frequency relation.

#### 3.3.2.6 Priority Handling for EN-DC

The NR frequency relations used by this feature can be prioritized to define

preferred frequencies when the UE searches for PSCell candidates.

Frequency priorities are controlled

through

the following attributes:

- Mcpc.endcLowPrioPSCellCandHandling
- McpcPSCellProfileUeCfg.lowHighFreqPrioClassification
- UeMCNrFreqRelProfileUeCfg.connModePrioPSCell

With these attributes, a frequency relation is defined as high-priority or

low-priority compared to other frequency relations.

The priority of frequency relations is checked when the priority period is active. During

the priority period, only high-priority measurement reports trigger a mobility action,

and low-priority reports are stored.

Only one

low-priority report can be stored for each UE at any given point of time. The

low-priority report that is stored for the UE is configured through the

Mcpc.endcLowPrioPSCellCandHandling attribute. A

stored low-priority report only triggers a mobility action in the following cases:

- The priority period ends.
- The UE reports critical coverage.

After the priority period ends, both high- and low-priority reports trigger a

mobility action. The priority period is started when the UE is configured to search for

cells on alternative NR frequencies.

The priority period is not used in the following cases:

- All frequencies are defined as high-priority frequencies.
- All frequencies are defined as low-priority frequencies.

Changing the low-priority frequency handling through the

Mcpc.endcLowPrioPSCellCandHandling attribute can affect the

number of UEs moved between different NR frequencies.

### 3.4.3 Intra-Frequency Mobility Control for NR SA

In NR SA, intra-frequency handover can be triggered when an A3 measurement report is

received from the UE.

The A3 measurement report indicates that a neighbor NR cell is offset better than

the serving cell.

More than one neighbor cell can be included in the A3 measurement report through

configuring the IntraFreqMCCellProfileUeCfg.maxReportCells attribute,

however, the gNodeB only examines the first reported cell.

When the A3 measurement report is received, the gNodeB checks where the NR cell is

located. The following alternatives exist:

- The neighbor NR cell is covered by a different gNodeB. The gNodeB initiates an inter-gNodeB handover. Handover occurs over the NG-C interface or over the Xn interface, with direct packet forwarding over the Xn-U interface.
- The NR cell is covered by the same gNodeB. The gNodeB initiates an intra-gNodeB handover.

Figure 13   Measurement Handling and Intra-Frequency Handover

The A3 measurements and intra-frequency handover are handled as follows:

1. The UE is within NR coverage, and it connects to an NR cell from idle mode or after a handover.
2. The UE is instructed by the gNodeB to perform an A3 measurement.
3. The UE detects at least one neighbor NR cell that is offset better than the serving cell. The A3 measurement is triggered in the UE, and an A3 measurement report is sent to the gNodeB.
4. The gNodeB checks if a handover is possible to the reported cell. Note: The NR Automated Neighbor Relations feature can be used to find a valid target for SA mobility. If the PCI evaluation or UE capability check fails, the gNodeB ignores the measurement report.

Conditional Xn-C setup is triggered if

the gNodeB receives the intra-frequency A3 measurement report, which includes the

reported cell has an external cell relation from NR to NR. The reported cell then

belongs to the unknown external gNodeB.

### 3.4.4 Mobility Control at Poor Coverage for NR Standalone

When the UE moves into poor coverage in the PCell, the UE can search for PCell

candidates on alternative NR or LTE frequencies.

The Mobility Control at Poor Coverage function handles the situation when the UE moves

into poor coverage on the current

serving

PCell frequency and needs to be assigned to another cell on another

frequency with better coverage.

The function introduces search zones along the edge of the cell, where the UE makes

measurements for alternative cells on other frequencies. If a suitable cell is detected,

a mobility action is initiated. If no alternative frequencies are detected in the search

zone and the UE moves into critical coverage, the UE is released either with or without

redirect to Idle Mode.

For a UE that encounters poor coverage, the function makes it possible to search for

several inter-frequency and IRAT target frequencies, and configure priorities between

them. The function supports up to three independent and simultaneous search zones:

- Downlink signal strength, based on UE measurements
- Downlink signal quality, based on UE measurements
- Uplink signal quality, based on internal gNodeB measurements

The following mobility actions are supported:

- Handover to an NR frequency
- IRAT inter-system handover to an LTE frequency
- Release with redirect to an LTE frequency

The function allows the operator to enable frequencies for mobility with the following

attributes:

- UeMCNrFreqRelProfileUeCfg.connModeAllowedPCell for NR frequencies
- UeMCEUtranFreqRelProfileUeCfg.connModeAllowedPCell for LTE frequencies

The function allows the operator to prioritize the frequencies for mobility with the

following attributes:

- Mcpc.nrLowPrioPCellCandHandling
- McpcPCellProfileUeCfg.lowHighFreqPrioClassification
- UeMCNrFreqRelProfileUeCfg.connModePrioPCell for NR frequencies
- UeMCEUtranFreqRelProfileUeCfg.connModePrioPCell for LTE frequencies

The trigger quantities to be used for the function can be configured with

the McpcPCellProfileUeCfg.mcpcQuantityList attribute. The

function supports measurement reports based on the following trigger quantities:

- Downlink RSRP
- Downlink RSRQ
- Downlink SINR
- Uplink SINR

The Mobility Control at Poor Coverage function is prioritized over traffic steering.

#### 3.4.4.4 Search Zone for Mobility Control at Poor Coverage

UEs in the search zone search for PCell candidates on other frequencies with better

coverage.

The following search zones can be used independently of each other:

- Downlink signal strength, based on UE measurements
- Downlink signal quality, based on UE measurements
- Uplink signal quality, based on internal gNodeB measurements

Search zones are configured to use the trigger quantity specified in

the McpcPCellProfileUeCfg.mcpcQuantityList attribute. Separate

search zone measurements are configured if more than one trigger quantity is specified.

When the UE is in a search zone, it is considered to be in poor coverage for the used

trigger quantity.

If there are NR inter-frequency or LTE frequency candidates on which the UE can search

for PCell candidates, the following measurements are used to detect transition to and

from search zone or into critical coverage:

This measurement is not deconfigured when transitioning to or from the search zone.

#### 3.4.4.5 Evaluation of Frequency Relations

The feature relies on a set of NR and LTE frequency relations that is used to decide

where the UE can make measurements to search for PCell candidates on other NR and LTE

frequencies.

The NR frequency selection is based on the following criteria:

- If the frequency relation only allows certain PLMNs and the UE has an MRL, then at least one PLMN must be common between the PLMN list and the MRL.
- The UE must be capable of measurement on the target frequency.
- The McpcPCellNrFreqRelProfileUeCfg.inhibitMeasForCellCandidate attribute must be set to false for the target frequency.
- The UeMCNrFreqRelProfileUeCfg.connModeAllowedPCell attribute must be set to true for the frequency relation.

The LTE frequency selection is based on the following criteria:

- If the frequency relation only allows certain PLMNs and the UE has an MRL, then at least one PLMN must be common between the PLMN list and the MRL.
- The UE must be capable of measurement on the target frequency.
- The target RAT is not restricted according to the MRL.
- The McpcPCellEUtranFreqRelProfileUeCfg.inhibitMeasForCellCandidate attribute must be set to false for the target frequency.
- The UeMCEUtranFreqRelProfileUeCfg.connModeAllowedPCell attribute must be set to true for the frequency relation.

#### 3.4.4.6 Priority Handling

The NR and LTE frequency relations used by this feature can be prioritized to define

preferred frequencies when the UE searches for PCell candidates.

Frequency priorities are controlled

through

the following attributes:

- Mcpc.nrLowPrioPCellCandHandling
- McpcPCellProfileUeCfg.lowHighFreqPrioClassification
- UeMCNrFreqRelProfileUeCfg.connModePrioPCell for NR frequencies
- UeMCEUtranFreqRelProfileUeCfg.connModePrioPCell for LTE frequencies

Through

these attributes, a frequency relation can be defined as high priority or low priority

compared to other frequency relations.

The priority of frequency relations is checked when the priority period is active. During

the priority period, only high-priority measurement reports trigger a mobility action,

and low-priority reports are stored.

Only

one low-priority report can be stored for each UE at any given point of time. The

low-priority report that is stored for the UE is configured through the

Mcpc.nrLowPrioPCellCandHandling attribute. A

stored low-priority report only triggers a mobility action in the following cases:

- The priority period ends.
- The UE reports critical coverage.

After the period ends, both

high-priority

and low-priority reports trigger a mobility action. The priority period is started when

the UE is configured to search for cells on alternative NR and LTE frequencies.

The priority period is not used in the following cases:

- All frequencies are defined as high-priority frequencies.
- All frequencies are defined as low-priority frequencies.

Changing the low-priority frequency handling through the

Mcpc.nrLowPrioPSCellCandHandling

attribute can affect the number of UEs moved from NR to LTE frequencies.

#### 3.4.4.7 Handover and Release with Redirect at Poor Coverage

The Mobility Control at Poor Coverage function handles inter-frequency and IRAT

mobility for UEs at poor coverage.

The following cases are supported for the Mobility Control at Poor Coverage function:

- Handover and release with redirect at poor coverage based on Downlink Evaluation
- Handover and release with redirect at poor coverage based on Uplink Evaluation

The function uses gNodeB-based uplink-quality measurements or UE-based downlink-quality

measurements to check if a PCell handover needs to be initiated, and to determine the NR

inter-frequency cell or the E-UTRAN cell that is a good candidate for PCell handover.

These measurements depend on the trigger quantity selected in the

McpcPCellProfileUeCfg.mcpcQuantityList attribute.

- When RSRP quantity is selected, downlink A1or A2 RSRP measurements are configured in the UE to detect the transitions to and from the downlink RSRP search zone and critical coverage. When in search zone, RSRP is used as trigger quantity to search for inter-frequency and IRAT PCell candidates with A5 and B2 measurements.
- When RSRQ quantity is selected, downlink A1 or A2 RSRQ measurements are configured in the UE to detect the transitions to and from the downlink RSRQ search zone and critical coverage. When in search zone, RSRQ is used as trigger quantity to search for inter-frequency and IRAT PCell candidates with A5 and B2 measurements.
- When downlink SINR quantity is selected, downlink A1 or A2 SINR measurements are configured in the UE to detect the transitions to and from the downlink SINR search zone and critical coverage. When in search zone, downlink SINR is used as trigger quantity to search for inter-frequency and IRAT PCell candidates with A5 and B2 measurements.
- When uplink SINR quantity is selected, uplink SINR measurements are configured internally in the gNodeB to detect the transitions to and from the uplink SINR search zone and critical coverage. When in search zone, RSRP is used as trigger quantity to search for inter-frequency and IRAT PCell candidates with A5 and B2 measurements.

#### 3.4.4.8 Additional Target Evaluation of Other Reported Quantity for Mobility Control at Poor Coverage

- McpcPCellProfileUeCfg.rsrpCandidateA5AddTargEval

### 3.4.5 Uplink Quality Evaluation at Incoming Handovers

The target gNodeB can perform an uplink quality-based evaluation at incoming handover

to prevent handover back

to

the source cell.

When the target gNodeB receives the NGAP HANDOVER REQUEST message or the

XnAP HANDOVER REQUEST message from a source gNodeB, the target gNodeB

can estimate the expected uplink quality for the UE in the target cell. The uplink

quality estimation is performed

if

the following conditions are all fulfilled in the target gNodeB:

- The value of the UeMCCellProfileUeCfg.checkUlQualityIncomingHo attribute is set to true.
- The source gNodeB provides a downlink RSRP value of the target cell in the NGAP HANDOVER REQUEST message or the XnAP HANDOVER REQUEST message.
- The value of the GNBCUCPFunction.resourceStatusReportF1Enabled attribute is set to true.

If any of the above conditions are not fulfilled, the uplink quality evaluation is

skipped at incoming handover.

Uplink quality is an estimation of the maximum value of the uplink SINR. Maximum uplink

SINR is a normalized uplink SINR value that assumes that

a

UE uses a single PRB in uplink. The estimation considers the sum of

the uplink noise and the interference in the target cell, the UE

Tx

power, and the estimated uplink path loss in the target cell. Updated SSB beamforming

gain after cell setup is not supported during uplink quality evaluation at incoming

handovers.

The UeMCCellProfileUeCfg.ulQualityThreshIncomingHo attribute defines

the minimum uplink quality of the target cell that is required to allow an incoming

handover. If the estimated uplink quality is equal to or larger than the threshold value

defined in the UeMCCellProfileUeCfg.ulQualityThreshIncomingHo

attribute, the incoming handover is allowed. If the estimated uplink quality is below

the threshold value, the incoming handover is rejected and the target gNodeB sends the

NGAP HANDOVER PREPARATION FAILURE or the XnAP HANDOVER

PREPARATION FAILURE message.

### 3.4.6 Release with Redirect to LTE at UE Failing Capability Check

The UE capability check procedure is performed at Initial Context Setup signaled over the

NG-C interface. If the UE capabilities are not received from the AMF or not stored in

the gNodeB, the gNodeB initiates the UE

capability inquiry by sending the RRC UECapabilityEnquiry message to the

UE. The UE responds with the RRC UECapabilityInformation message that

contains all the capability information the network requested. Based on the evaluation

of the received UE capability information, the gNodeB either

reconfigures the UE to provide the most optimal radio configuration, or initiates the

release of the UE to RRC\_IDLE mode. The latter happens when the UE

fails the capability check by not supporting the minimum capabilities required by the

network.

When the UE is released to RRC\_IDLE mode, it is redirected to an LTE frequency.

The gNodeB selects the target LTE redirection candidates based on the configuration of

the UeMCEUtranFreqRelProfileUeCfg.blindRwrAllowed attribute. If the

target frequency band is not supported by the UE or the target PLMN of this frequency is

not allowed according to the mobility restriction list, this redirection candidate is

considered not valid. In this case, the gNodeB checks the next possible redirection

candidate according to the ascending order of the ARFCN. If no more redirection

candidates are configured, the UE is released to idle mode. The

UeMCEUtranFreqRelProfileUeCfg.blindRwrAllowed attribute can be

set to true only for frequency relations where the

UeMCEUtranFreqRelProfileUeCfg.connModeAllowedPCell attribute is

set to true.

If no suitable LTE frequency is found, or no LTE coverage is available, the UE remains in RRC\_IDLE mode.

It is possible to configure the maximum number of simultaneous event measurements for

inter-frequency event measurements on non-serving NR frequencies.

For more information about the minimum requirements related to measurements, see

3GPP TS 38.133; Requirements for support of radio resource

management.

By default, the Secondary Node follows the 3GPP-defined minimum UE requirements, and it

limits the number of measurements that it requests the UE to perform. It is possible to

change the default Secondary Node behavior to allow the Secondary Node to increase the

number of simultaneous measurements it requests the UE to perform.

the default value is not enough to support all mobility and traffic steering functions

and their need for event-based measurements. The functions are served in the order in

which they request measurements, up to the limit defined by the related attribute. To

support deployments with a high number of frequency layers, the related attribute has a

value range that makes it possible to exceed the 3GPP-defined minimum requirement.

## Appendix A.m: 21.Q4: Cell Reselection Priority at UE Release

This feature enhancement enables sending CRP to UEs when they are released. As a

result, while UEs are in idle mode, they are kept in their last PCell and connected to NR as

long as carrier aggregation is configured.

| Feature Name:           | NR Mobility         |
|-------------------------|---------------------|
| Feature Identity:       | FAJ 121 5041        |
| Value Package Name:     | NR Base Package     |
| Value Package Identity: | FAJ 801 4002        |
| Node Type:              | Baseband Radio Node |
| Access Type:            | NR                  |

Summary and Benefits

For carrier aggregation, the Capability-Aware PCell Selection function might move UEs

to a PCell where they have the best estimated downlink throughput. However, when a

UE enters idle mode, SIB might direct it to a less optimal cell than its current

PCell. When such a UE reenters connected mode, it needs to be moved again.

If this feature enhancement is enabled, the gNodeB sends CRP at UE release if the

following criteria are met:

- The UE is configured with carrier aggregation.
- The UE is released from connected mode to idle mode because of inactivity.

CRP overrides the SIB priority to keep idle mode UEs in their current PCell. This

benefits UEs that are already in a cell that is most suitable to their capabilities.

When such a UE reenters connected mode, no additional handover is required because

of the Capability-Aware PCell Selection function. This benefit applies to stationary

UEs especially.

An additional benefit is that UEs are kept on NR frequencies as long as carrier

aggregation is configured, because the CRP used by this feature enhancement

prioritizes NR frequencies over LTE frequencies.

Capacity and Performance

System performance is improved because the number of handovers is reduced.

Operation

To enable this feature enhancement, the

TrStSaCellProfileUeCfg.capAwarePCellSelEnabled attribute must

be set to true and the

TrStSaCellProfileUeCfg.actionAtRelease must be set to

CA\_CONF.

The following MO attributes are introduced:

- TrStSaCellProfileUeCfg.actionAtRelease
- TrStSaCellProfileUeCfg.t320

The NRCellCU.pmEbsUeCtxtRelNormalTsCaConf EBS counter is

introduced to observe how often traffic steering is performed at UE release due to

this feature enhancement. This counter uses the CuCpProcUeCtxtRel event, which is

updated with a parameter to support this counter.

The range of the TrStSaCellProfileUeCfg.criterionBetterUeConfig

attribute is extended.

Interfaces

the UE:

- CRP
- The timer set with the TrStSaCellProfileUeCfg.t320 attribute

For more information on CRP and the t320 timer, see 3GPP TS

38.331.

## Appendix A.ak: 24.Q4.0: Uplink-Aware Mobility for NR Standalone

This feature enhancement provides uplink quality-based support for Mobility Control

at Poor Coverage in NR standalone.

| Feature Name           | NR Mobility         |
|------------------------|---------------------|
| Feature Identity       | FAJ 121 5041        |
| Value Package Name     | NR Base Package     |
| Value Package Identity | FAJ 801 4002        |
| Node Type              | Baseband Radio Node |
| Access Type            | NR                  |
| Impact at Upgrade      | Optional            |

Summary and Benefits

This feature enhancement introduces support for uplink-aware inter-frequency and IRAT

handover based on uplink quality measurements in NR. The uplink-aware mobility is

performed when poor uplink quality is detected in the source cell. To prevent

handover back to the source cell, uplink quality is evaluated during incoming

handover in target cell.

This feature enhancement has the following benefits:

- Improved configuration flexibility for handover border tuning when downlink and uplink coverage varies.
- Improved integrity KPIs by reducing UE throughput degradation.
- Improved retainability KPIs by reducing the number of dropped calls.

Capacity and Performance

When poor uplink quality is evaluated at incoming handover, the handover is rejected

resulting in handover preparation failure. This leads to a degradation in Mobility

KPIs. This happens because the NRCellCU.pmHoPrepAttOutInterGnb PM

counter is stepped, but the NRCellCU.pmHoPrepSuccOutInterGnb PM

counter is not stepped.

NG-based and Xn-based handovers might fail because of poor uplink quality in the

target NR cell. The following counters can e used to monitor the mobility KPI

degradation:

- NRCellCU.pmHoPrepFailInInterGnbPoorUl
- NRCellCU.pmEbsHoPrepFailInInterGnbPoorUl
- NRCellCU.pmHoPrepFailOutInterGnbTargetPoorUl
- NRCellCU.pmEbsHoPrepFailOutInterGnbTargetPoorUl

The existing CuCpProcNrHoPrepOut and CuCpProcNrHoPrepIn PM events are updated

with new HoPrepResult values for the purpose.

The number of RRC reconfiguration messages is expected to increase. This is because

uplink-aware Mobility Control at Poor Coverage adds the option for A5 and B2

measurement configuration using RSRP as the trigger quantity through RRC when poor

uplink quality is detected. The measurement configuration is performed independently

from the A5 and B2 measurement configuration for downlink Mobility Control at Poor

Coverage, up to two instances of which can be configured. Therefore, three instances

of Mobility Control at Poor Coverage based on each trigger quantity can be used

simultaneously for uplink-aware mobility.

The number of RRC measurement events is expected to increase if three Mobility

Controls at Poor Coverage processes are used at the same time.

When uplink-aware Mobility Control at Poor Coverage receives an RSRP-based A5 or B2

measurement report, the other reported downlink quantities of the target cell are

verified before triggering handover. The CuCpMeasurementReportSaEval event can be

used to monitor the impact on RRC signaling load.

F1AP signaling load is expected to increase. This is because the uplink measurement

report is sent through F1AP messages when the event occurs in the node. The new

NRCellCU.pmMeasReportUlSa andNRCellCU.pmEbsMeasReportUlSa counters and the

CuCpUlMeasurementReportSaEval event can be used to monitor the impact on F1AP

signaling load.

Operation

Based on uplink coverage, the UE uplink measurements are configured in the node to

monitor that poor uplink quality is detected when the UE connects to the cell. The

UE uplink measurements are configured and the mobility actions are performed based

on the following measurement reports:

- When the node reports the A2 UL Entering Search Zone measurement, an A5 or B2 measurement, or both start for the UE to find a better cell. When the node receives an A5 or B2 measurement report, an inter-frequency or IRAT handover is triggered.
- When the node reports the A1 UL Leaving Search Zone measurement, the A5 or B2 measurement is removed.
- When the node reports the A2 UL Critical Coverage measurement, the UE is released, or a blind RwR is triggered.
