# 1 LTE-NR Dual Connectivity Overview

| Feature Name           | LTE-NR Dual Connectivity                                |
|------------------------|---------------------------------------------------------|
| Feature Identity       | FAJ 121 4908                                            |
| Value Package Name     | NR Base Package                                         |
| Value Package Identity | FAJ 801 4002                                            |
| Node Type              | Baseband Radio Node                                     |
| Access Type            | NR                                                      |
| Licensing              | Non-license-controlled feature. No license is required. |

Summary

The LTE-NR Dual Connectivity (FAJ 121 4908) feature

introduces support for E-UTRA - NR Dual Connectivity (EN-DC) in the gNodeB being the

Secondary Node in the EN-DC deployment.

EN-DC is realized as the combination of the following features and system

functions:

- Basic Intelligent Connectivity (FAJ 121 4843)
- LTE-NR Dual Connectivity (FAJ 121 4908)
- LTE carrier aggregation
- NR carrier aggregation

With the LTE-NR Dual Connectivity feature activated, SCG radio resources provided by the

Secondary Node (gNodeB) in the EN-DC deployment can be set up for the EN-DC-capable

UE in NR coverage.

The benefits of the EN-DC deployment are the following:

- Higher peak rate of network data traffic by aggregating LTE and NR carriers.
- Sustainable capacity and performance growth.

Additional Information

For more information on EN-DC deployment, see the following documentation:

- 3GPP TS 37.716 specifications "E-UTRA - NR Dual Connectivity of x bands"

Hardware Requirements

The LTE-NR Dual Connectivity feature is supported on all Baseband units supporting

NR.

Limitations

- Both the Master Node and the Secondary Node must be time and phase synchronized either using GNSS or the IEEE 1588 Time and Phase Synchronization feature. A less strict accuracy of 500 Î¼s can be used with the Partial Timing Support (PTS) with relaxed network limits enhancement. In addition, both of the nodes have to have the same frame start offset. For more information, see Manage Network Synchronization. If EN-DC is deployed without the time synchronization feature, EN-DC setup is only guaranteed between LTE cells and NR cells if either of the following conditions is met: Both the LTE cells and the NR cells are on the same Baseband Radio Node. The LTE cells and the NR cells are synchronized by the means of Node Group Synchronization. If the node is deployed without the IEEE 1588 Time and Phase Synchronization feature, the Sync Time and Phase Accuracy Too Low alarm is raised. Because of this, it is recommended to activate the IEEE 1588 Time and Phase Synchronization feature in EN-DC deployments.
- Intra-band carrier aggregation in EN-DC is only supported with the following 3GPP operating bands: B41 for LTE n41 for NR Note: Intra-band carrier aggregation is blocked when the Master Node is set to NR-preferred carrier aggregation and to block intra-band EN-DC.
- Packet forwarding is not supported for Secondary Node terminated DRBs during RRC connection re-establishment.
- For LTE mobility without PSCell change, the following limitations apply: No bearer type transition occurs during handover. SGW relocation for Secondary Node-terminated DRBs results in Secondary Node release.
- LTE uplink carrier aggregation is not used in EN-DC.

Network Requirements

- The LTE-NR Dual Connectivity feature requires the S1AP E-RAB Modification Indication procedure to be supported in the EPC.
- The RLC mode for a specific QCI defined in the Master Node has to reflect the corresponding setting in the Secondary Node. For the Secondary Node, the RLC mode for a QCI is set using the CUCP5qi.rlcMode attribute for the corresponding 5QI in the CUCP5qiTable MO instance. This MO instance must have the CUCP5qiTable.default5qiTable attribute set to true if the CUCP5qi.profile5qi attribute has a value configured that is equal to either of the following attribute values set on the Master Node: QciProfileOperatorDefined.qci QciProfilePredefined.qci

# 3 Feature Operation of LTE-NR Dual Connectivity

## 3.1 EN-DC Deployment

EN-DC, covering the fundamental interaction between a Master Node (eNodeB) and a Secondary Node (gNodeB),

is realized as the combination of the following features and system functions:

- Basic Intelligent Connectivity
- LTE-NR Dual Connectivity
- LTE carrier aggregation
- NR carrier aggregation

The EN-DC-capable UE is always connected to the eNodeB, being

the Master Node in EN-DC. The UE can connect to the gNodeB,

being the Secondary Node in EN-DC, if the UE is within NR coverage. The UE is connected

both to the eNodeB and the gNodeB

through the Uu interface.

The eNodeB connects to the EPC through the S1 interface. The

S1-C interface is used for control plane, while the S1-U interface is used for user

plane.

The gNodeB connects to the EPC through the S1-U interface for

user plane.

The eNodeB and the gNodeB are connected

to each other through the X2 interface. The X2-C interface is used for control plane,

while the X2-U interface is used for user plane. One eNodeB can

be connected to several gNodeBs and one gNodeB can be connected to several eNodeBs.

The UE uses either of the following combinations of radio resources when connected to

both nodes in EN-DC:

- Master Node terminated MCG DRBs and Secondary Node terminated split DRBs Such a case applies when the UE is in NR coverage and can use both MCG and SCG resources depending on the QCI configuration of the DRBs. Split is allowed for DRBs by configuration.
- Master Node terminated MCG DRBs and Secondary Node terminated MCG DRBs Such a case applies when the UE is only served by the MCG because of an SCG Release procedure. However, Secondary Node terminated MCG DRBs are also used when DRBs are not allowed to be split by configuration.

In EN-DC, the UE is in RRC\_CONNECTED mode in the eNodeB. The control of the UE is managed through the LTE RRC

signaling between the eNodeB and the UE. Information specific

to gNodeB resources is carried by containers conveying NR RRC

signaling over the X2-C and the LTE Uu interfaces.

The gNodeB resources, including a candidate NR cell, for the

EN-DC-capable UE are configured using the following procedures:

- Secondary Node Addition
- Secondary Cell Group Addition

The eNodeB selects a candidate NR cell using one of the

following methods:

| Configuration-based setup   |    | One candidate NR cell is referenced by the primary LTE cell serving the                         UE.                     |
|-----------------------------|----|-------------------------------------------------------------------------------------------------------------------------|
|                             |    |                                                                                                                         |
| Measurement-based setup     |    | The eNodeB orders the UE to perform an LTE inter-RAT                         B1 measurement to find a suitable NR cell. |

With the gNodeB resources available to the EN-DC-capable UE, DRBs

can be reconfigured to become Secondary Node terminated split DRBs. With the DRBs

reconfigured to be split, the user plane provided to the UE is configured in the

following ways:

- Downlink traffic is transmitted through the gNodeB using SCG resources after a successful NR random access procedure. If the NR link quality falls below a configured threshold, downlink traffic is redirected to use MCG resources for the split DRB. Initial downlink transmission is done through SCG. However, the further use of SCG, MCG, or the combination of the two is determined by NR link quality and buffer age.
- Uplink traffic is transmitted either through the eNodeB using MCG resources or the gNodeB using SCG resources. The initial uplink transmission is configurable. However, the NR link quality determines the further use of MCG or SCG resources for uplink traffic.

In both directions, the Secondary Node terminated split DRB uses NR PDCP resources.

## 3.2 Scenarios Triggering EN-DC Procedures in the Secondary Node

### 3.2.1 Secondary Node User Activity Supervision

The Secondary Node is responsible for monitoring user plane activity on the Secondary

Node terminated DRBs serving each UE.

When no user plane data is sent to or received from the UE on any of the Secondary Node

terminated DRBs for the amount of time configured through the

InactivityProfileUeCfg.tInactivityTimerEndcSn attribute, the

Secondary Node sends the X2AP SGNB ACTIVITY NOTIFICATION message to the

Master Node. The message indicates to the Master Node that the UE is inactive on

Secondary Node terminated DRBs.

If user plane data is sent to or received from the UE on any of the Secondary Node

terminated DRBs after the inactivity notification, the Secondary Node sends an

additional X2AP SGNB ACTIVITY NOTIFICATION message to the Master Node.

The message indicates to the Master Node that the UE is reactivated.

For more information on user activity supervision, see NR Service-Adaptive Inactivity Timer.

The Secondary Node initiates the Secondary Node Release procedure when it detects that

the configured maximum number of allowed RLC retransmissions is reached in downlink.

The Secondary Node initiates the Secondary Node Release procedure when the state of the

- The NRCellDU.administrativeState attribute is set to LOCKED by the operator.
- The NRCellDU.operationalState attribute is set to DISABLED by an underlying fault in the referenced sector carrier or by sector carrier lock.

### 3.2.5 PDCP COUNT Threshold Is Reached

The Secondary Node initiates the Secondary Node Release procedure when the uplink or

downlink PDCP COUNT threshold is reached for at least one Secondary

Node terminated DRB (both split and MCG).

The purpose of the release is to avoid COUNT wrap-around.

The UAI overheating report is forwarded by the Master Node to the Secondary

If the Secondary Node receives the X2AP SGNB MODIFICATION REQUEST message

## 3.3 EN-DC Bearer Type Transitions

Bearer type transitions in EN-DC are possible among the following DRB types:

- Master Node terminated MCG DRBs
- Secondary Node terminated split DRBs
- Secondary Node terminated MCG DRBs

The EN-DC-configured UE can be in one of the following states, depending on the

combination of the DRB types, as shown in Figure 3:

Table 2   EN-DC-Configured UE States with DRB Type Combination Used

| EN-DC-Configured UE State   | Combination of DRB Types                                                                                                                                         |
|-----------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| A                           | This combination only contains Master Node terminated MCG DRBs.                                                                                                  |
| B                           | This combination contains one or more Secondary Node terminated                                     split DRBs and zero or more Master Node terminated MCG DRBs. |
| C                           | This combination contains one or more Secondary Node terminated                                     MCG DRBs and zero or more Master Node terminated MCG DRBs.   |

Figure 3   Bearer Type Transition in EN-DC

1. UE Connection Setup The EN-DC-capable UE is connected only by using Master Node terminated MCG DRBs. Therefore, in the UE Connection Setup, user plane data is transmitted through the Master Node only.
2. Secondary Node Addition During the Secondary Node Addition procedure, one or more of the existing Master Node terminated MCG DRBs are reconfigured to Secondary Node terminated split DRBs.
3. SCG Release (Secondary Node Modification) During SCG Release, all Secondary Node terminated split DRBs are reconfigured to the Secondary Node terminated MCG DRBs. The SCG resources for the UE are released, but the PDCP context is kept in the Secondary Node.
4. SCG Addition (Secondary Node Modification) During SCG Addition, all Secondary Node terminated MCG DRBs are reconfigured to Secondary Node terminated Split DRBs.

All SCG resources for the UE are released. PDCP resources for the UE are relocated from the Secondary Node to the Master Node.

The UE can be released to RRC\_IDLE mode from any of the EN-DC-configured

UE states.

## 3.4 EN-DC Configuration

### 3.4.1 Dynamic Power Sharing

The uplink transmission power capability of the EN-DC-capable UE must be split between

the SCG radio resources and the MCG radio resources when the UE is configured with at

least one Secondary Node terminated split DRB. This is required if the UE is using an

EN-DC band combination with an NR band in Low-Band or Mid-Band.

If the UE is capable of dynamic power sharing on the used EN-DC band combination, the UE

manages its transmission power capability by itself. In Low-Band and Mid-Band, dynamic

power sharing allows the UE to configure the maximum uplink transmission power for both

MCG and SCG radio resources in steps of 1 dB.

If user data transmission for both MCG and SCG radio resources is scheduled at the same

time, transmission power is allocated in the following order:

1. Power is allocated first to the MCG radio resources.
2. The remaining power is allocated to the SCG radio resources.

The allocation of uplink transmission power ensures that the total uplink power

configured separately for the MCG and SCG radio resources is not exceeded.

If the UE is not capable of dynamic power sharing, the network needs to configure the UE

with a maximum uplink transmission power semi-statically for both the SCG and the MCG

radio resources.

The Secondary Node receives the maximum uplink transmission power for

the SCG resources over the X2 interface.

For more information on dynamic power sharing and UE power classes,

see the following documentation:

- 3GPP TS 38.101, NR; User Equipment (UE) Radio Transmission and Reception; Release 15
- 3GPP TS 38.331, NR; Radio Resource Control (RRC); Release 15

### 3.4.2 Switching of Downlink MCG and SCG Radio Resources of the Secondary Node Terminated Split DRB

The switching of downlink MCG and SCG radio resources of the Secondary Node terminated

split DRB allows downlink user plane transmission to conform to the estimated radio

quality of the NR link. The switching depends on the following conditions:

- When the estimated radio quality of the NR link is good, the downlink data transmission uses the SCG radio resources.
- When the estimated radio quality of the NR link is poor, the downlink data transmission uses the MCG radio resources.

The switching of downlink MCG and SCG resources can be adapted to rapidly changing radio

conditions. This is possible because it does not require the reconfiguration of the UE.

The trigger values for switching downlink MCG and SCG radio resources are

configurable.

To enable MCG and SCG radio resource switching in downlink, the

CgSwitchCfg.dlCgSwitchMode attribute must be set to

ENABLED in the serving NR cell.

The switching of downlink MCG and SCG resources is based on repeated

polling for estimated downlink radio quality and periodicity. The switching can be

configured for Low-Band and Mid-Band using the

CgSwitchCfg.dlScgNoDataAcsiPeriodicity attribute. To disable the

polling and MCG and SCG radio resource switching in downlink, the

CgSwitchCfg.dlCgSwitchMode attribute must be set to

false in the serving NR cell.

On High-Band, if switching happens from SCG to MCG due to poor radio conditions, two

consecutive CQI reports indicating good radio quality are needed until the switch back

to SCG is performed.

It is recommended to enable the NR Coverage Triggered Secondary Node Release function if

the switching of downlink MCG and SCG resources is disabled.

### 3.4.3 Switching of Uplink MCG and SCG Radio Resources of the Secondary Node Terminated Split DRB

The switching of uplink MCG and SCG radio resources of the Secondary Node-terminated

split DRB allows uplink user plane transmission to conform to the estimated radio

quality of the NR link. The switching depends on the following conditions:

- When the estimated radio quality of the NR link is good, the uplink data transmission uses the SCG radio resources.
- When the estimated radio quality of the NR link is poor, the uplink data transmission uses the MCG radio resources.

To enable MCG and SCG radio resource switching in uplink, the

UlCgSwitchCfg.ulCgSwitchMode attribute must be set to

ENABLED in the serving NR cell.

The switching of MCG and SCG radio resources in uplink is slower than it is in downlink

because it involves the reconfiguration of the

UE.

The configuration of the

UlDataSplitUeCfg.initialUlPrimaryPathMode

attribute determines whether the MCG or the SCG radio resources of the Secondary

Node-terminated split DRB is used initially for user plane transmission in uplink.

When

the Service-Adaptive Uplink User Plane Profile feature is activated, different

configurations can be used for each 5QI, which can be configured with the

CUCP5qi.endcUlDataSplitRef and the

CUCP5qi.nrdcUlDataSplitRef attributes, for EN-DC and NR-DC,

respectively. The CUCP5qi.endcUlDataSplitRef attribute references a

UlDataSplit MO instance, which is the parent MO class of the

UlDataSplitUeCfg.initialUlPrimaryPathMode attribute. If the

Service-Adaptive Uplink User Plane Profile feature is deactivated, the following MO

instances are applied for all CUCP5qi MO instances for EN-DC and

NR-DC, respectively, instead of the referenced instances:

- UlDataSplit=Default\_endc
- UlDataSplit=Default\_nrdc

For more information, see Service-Adaptive Uplink User Plane

Profile.

The trigger values for switching uplink MCG and SCG radio resources are configurable.

The following attributes can be used to configure trigger values for switching between

uplink MCG and SCG radio resources:

- UlCgSwitchCfg.ulScgLowQualThresh
- UlCgSwitchCfg.ulScgLowQualHyst

The radio quality of the NR link is defined by the threshold value

configured

through the

UlCgSwitchCfg.ulScgLowQualThresh attribute. If the NR radio quality

drops below the configured threshold value, the gNodeB switches to the MCG radio

resources. The maximum value of the uplink SINR is estimated at every PUSCH reception. A

filter smoothes the uplink SINR samples. The smoothing is needed to normalize the

reaction time to temporary changes of the SINR. The maximum uplink-normalized SINR value

is checked against the value of the attribute.

In Low-Band and Mid-Band, if the UE quickly moves out of the NR cell coverage, the

detection of CRC failures and discontinuous transmissions in uplink causes the gNodeB to switch to the MCG radio resources. This happens

regardless of the estimated radio quality of the NR link.

In High-Band, an uplink radio resource switch is triggered if radio quality drops below

the configured

threshold,

or based on the failure of decoding CSI reports. If too many consecutive CSI CRC

failures are detected, the switch to MCG radio resources is triggered. When the radio

link quality improves, the

switch

back to SCG radio resources is triggered.

In High-Band, it is possible to switch to the DFTS-OFDM waveform and to a single uplink

TX port while user plane data is transmitted using SCG radio resources. This way, uplink

TX port switching provides a suitable buffer before switching to transmitting user plane

data using MCG radio resources.

The quality of MCG radio resources is not evaluated before the uplink resource switching.

The good quality of MCG radio resources must be ensured by the mobility procedures

determined by

the Master

Node-terminated MCG DRBs. Different uplink channel quality conditions

use different uplink switching mechanisms if both layer switching and radio resource

switching are enabled.

### 3.4.4 EN-DC Band Combination Coordination in the Secondary Node

The EN-DC band combination coordination function is used to

set up carrier aggregation (CA) in the MCG and the SCG. The Master Node is always the

eNodeB, and the Secondary Node is the gNodeB. The band combination coordination is

triggered to add the SCG during the following EN-DC procedures:

- Secondary Node Addition
- Secondary Node Modification

EN-DC band combination coordination can be set to LTE\_PREFERRED or

NR\_PREFERRED in the Master Node. Selecting the

LTE\_PREFERRED option optimizes LTE CA first, and maximizes CA

throughput for LTE. The NR\_PREFERRED option optimizes NR CA first and

then LTE CA. The NR\_PREFERRED option maximizes CA throughput for NR and

it is the recommended configuration for EN-DC band combination coordination.

The gNodeB prevents intra-band EN-DC if the Master Node is set to NR-preferred and blocks

intra-band EN-DC. This prevents considering frequencies in the same or overlapping

operating bands for EN-DC band combination coordination. The blocking has the following

effects:

- PSCell is selected based on the candidates the eNodeB sends to the gNodeB.
- PSCell is selected so that it does not conflict with the PCell band.
- NR SCells are not considered for CA if in the same band as the PSCell or the PCell.

Frequency-consistency check to avoid collision is performed by the Secondary Node at the

following steps:

- Secondary Node Addition and initial NR SCell selection
- NR SCell reselection
- PSCell change (and optional NR SCell change)
- Master Node Initiated Secondary Node Modification

The gNodeB rejects the EN-DC band combination in case of intra-band frequency collision

if it is blocked by the Master Node and there are no more PSCell candidates.

1. The Secondary Node receives a list of the allowed EN-DC band combinations from the Master Node. The band combinations match the selected LTE CA part and the target NR PSCell band. This list aims to maximize LTE throughput first. If EN-DC band combination coordination is set to NR\_PREFERRED, the Secondary Node receives a list of all band combinations that are valid for EN-DC CA. Band combinations are listed in a descending order according to maximized LTE throughput.
2. The Secondary Node selects the band combination and the feature set that are appropriate to the network configuration and that can maximize aggregated downlink bandwidth. If the UE capabilities do not allow the Secondary Node to select any band combinations, the request form the Master Node is rejected. If EN-DC band combination coordination is set to NR\_PREFERRED, the Secondary Node chooses the first available combination that maximizes NR throughput.
3. The selected band combination and feature set are signaled back to the Master Node, which completes the UE configuration.

For more information on Band Combination Coordination, see Basic Intelligent

Connectivity.

## 3.5 EN-DC Procedures

### 3.5.1 Secondary Node Addition

The Secondary Node Addition procedure is used to establish a UE context at the Secondary

Node which, this way, provides SCG radio resources to the EN-DC-capable UE.

Downlink packet forwarding is supported in EN-DC during the

Secondary Node Addition procedure. It is always enabled and not controlled by MO

configuration. Downlink packets are forwarded from the Master Node to the Secondary Node

using the X2-U interface.

During the Secondary Node Addition procedure, the EN-DC band combination is chosen in the

Secondary Node in coordination with the allowed EN-DC band combinations signaled by the

Master Node. The EN-DC-capable UE is configured with the MCG and SCG information (LTE

and NR carriers) according to the chosen EN-DC band combination.

Figure 4   Secondary Node Addition Procedure

Secondary Node Addition is not performed if the following

conditions are met:

- The PSCell is barred or reserved for the operator.
- The UeMCCellProfile and the UeMCCellProfileUeCfg MO classes are configured to specify that cells that are barred or reserved for the operator are also blocked for UEs in RRC\_CONNECTED mode.

1. Secondary Node Addition Preparation
    1. Following the configuration-based or the B1 measurement-based triggering of the Secondary Node Addition procedure, the Master Node sends an X2AP SGNB ADDITION REQUEST message to the Secondary Node through the X2-C interface. The message contains the EN-DC band combinations and an NR target cell the Master Node selected. A timer supervising EN-DC procedures over the X2 interface starts when the Master Node sends the message.
    2. If the Secondary Node sends the X2AP SGNB ADDITION REQUEST ACKNOWLEDGE message to the Master Node, new allocated SCG radio resources exist on the Secondary Node, as well as MCG radio resources on the Master Node, for the UE. The timer supervising EN-DC procedures over the X2 interface stops when the Master Node receives the message. The Secondary Node chooses one of the band combinations provided by the Master Node, and selects one PSCell, and possibly also NR SCells. The Secondary Node then sends the selected band combination back in the SGNB ADDITION REQUEST ACKNOWLEDGE message. The EN-DC configuration is finalized in the Master Node based on the selected band combination. Note: If the target PSCell is overloaded, the Secondary Node sends the X2AP SGNB ADDITION REQUEST REJECT message to the Master Node. After receiving this message, the Master Node includes the target PSCell in the NR cell denylist and excludes it from the Secondary Node Addition procedure. If the target PSCell is blocked for UEs in RRC\_CONNECTED mode, the Secondary Node sends the X2AP SGNB ADDITION REQUEST REJECT message to the Master Node with the cell-not-available or the target-not-allowed cause value. After receiving this message with the cell-not-available cause value, the Master Node includes the target PSCell in the NR cell denylist and excludes it from the Secondary Node Addition procedure.
    3. The Master Node sends the SN STATUS TRANSFER message to the Secondary Node. The SN STATUS TRANSFER message contains the PDCP status information (uplink and downlink COUNT values) for RLC AM bearers to be reconfigured from Master Node terminated to Secondary Node terminated DRBs.
    4. Separate LTE carrier aggregation thresholds are used for the split DRBs to control the activation and deactivation of LTE SCells. The following attributes are used to set the carrier aggregation thresholds on the Master Node: CarrierAggregationFunction.dcSCellActDeactDataThres CarrierAggregationFunction.dcSCellActDeactDataThresHyst CarrierAggregationFunction.dcSCellDeactDelayTimer
2. RRC Connection Reconfiguration
    1. The Master Node initiates the RRCConnectionReconfiguration message to the UE. The message includes the NR RRC configuration message the Master Node received from the Secondary Node.
    2. The UE applies the new configuration. LTE Random Access procedure between the UE and the Master Node is performed.
    3. The UE sends the RRCConnectionReconfigurationComplete message to the Master Node. All the LTE SCells are activated once the RRC reconfiguration of the UE is completed to receive the data.
3. The Master Node sends the SGNB RECONFIGURATION COMPLETE message to indicate to the Secondary Node that the UE is reconfigured successfully. The time to wait for the SGNB RECONFIGURATION COMPLETE message is configured in the GNBCUCPFunction.tDcOverall attribute on the Secondary Node. Note: It is recommended that the value of the GNBCUCPFunction.tDcOverall timer in the Secondary Node be greater than the value of the Rrc.tRrcConnectionReconfiguration timer in the Master Node.
4. Path Update
    1. The Master Node starts updating the user plane path for the Secondary Node terminated split DRBs by sending the E-RAB MODIFICATION INDICATION message to the MME.
    2. The MME confirms the user plane path update by sending the E-RAB MODIFICATION CONFIRM message to the Master Node.
5. NR Random Access procedure between the UE and the Secondary Node is performed.

#### 3.5.1.1 Packet Forwarding at Secondary Node Addition

Downlink packet forwarding is supported in EN-DC during the

Secondary Node Addition procedure. It is always enabled and not controlled by MO

configuration. Downlink packets are forwarded from the Master Node to the Secondary Node

using the X2-U interface.

During the Secondary Node Addition procedure, bearer resources allowed to be Secondary

Node terminated Split DRBs are moved from the Master Node to the Secondary Node.

Downlink packets are sent from the SGW to old bearer resources in the Master Node. During

the Secondary Node Addition procedure, a packet forwarding tunnel between the two nodes

is established over the X2-U interface.

Packet forwarding behavior between the old and the new bearer resources varies based on

the following conditions:

- In both RLC AM and UM, downlink packets that are not yet sent to the UE by the old bearer resources in the Master Node are forwarded to the newly set up Split DRBs. The Split DRBs in the Secondary Node send these packets to the UE.
- In RLC AM, downlink packets from the Master Node that are sent to, but not acknowledged by the UE, are forwarded to the Secondary Node terminated Split DRB where they are dropped.

Packet forwarding between old and newly set up bearer resources ensures that packet loss

is minimized during the Secondary Node Addition procedure.

Packet forwarding ends when the path switch is completed and the SGW starts sending

packets to the new bearer resources in the Secondary Node.

## 8.2 X2 Link Break

X2 link break in EN-DC occurs when the connection over the X2 interface is lost between

the Master Node (eNodeB) and the Secondary Node (gNodeB).

An X2 link break is typically caused by the following:

- The TermPointToENodeB MO on the Secondary Node to the Master Node is locked
- The termination point on the Master Node to the Secondary Node is locked
- Transport failure
- The Master Node restarts or is not operational

An X2 link break in EN-DC is detected in the Master Node and the Secondary Node

independently. The Secondary Node can lose connection to all Master Nodes over the X2

interface in EN-DC, and in such cases, the LTE Anchor Service

Unavailable alarm is raised on the Secondary Node. The alarm is

ceased if at least one of the following conditions is met:

- Connection to at least one Master Node is restored.
- All connections are manually locked on the Secondary Node side.
- All TermPointToENodeB MOs are deleted as described in Removal of TermPointToENodeB MO in X2 Configuration.

The Secondary Node releases all the resources associated with the

affected X2 link.

It is recommended to use optical 10G connection for all routers between the gNodeB and eNodeB in EN-DC if the LTE

air interface capacity is above 600 Mbps across all cells in the eNodeB.

X2AP partial reset support in EN-DC deployment allows the release of only

some of the UE using the X2 link. On a high-level, the process consists of the following

steps:

1. The Secondary Node sends the PARTIAL RESET REQUIRED message to the Master Node.
2. The Master Node responds with the PARTIAL RESET CONFIRM message.
3. All resources used by the UE indicated in the PARTIAL RESET REQUIRED message are released. To trigger the release, the same procedure is used as in the case of an X2 link break. The following messages are sent by the Master Node: S1 RESET to the MME RRCConnectionRelease is sent to the affected UE.

## 8.6 Configure Radio Link Failure Parameters for the Secondary Cell Group

In EN-DC, Radio Link Control (RLC) and Radio Resource Control (RRC) parameters exist

for SCG radio configuration. By using these parameters, the Radio Link Failure (RLF)

conditions can be set for the SCG resources available for the UE. The RLF conditions

configured using these parameters are signaled to the UE in the

RRCConnectionReconfiguration NR RRC message. If the UE detects

RLF, which means RLC or RRC parameter values are exceeded, it sends the

SCGFailureInformationNR LTE RRC message to the Master Node. The

purpose of the message is to indicate a failure of the SCG resources to the Master

Node.

At the reception of the SCGFailureInformationNR message, the Master

Node initiates the Secondary Node Release procedure.

Steps

1. Set the Rrc.t310 attribute to configure a waiting time for RLF. By configuring a waiting time for RLF, the trigger to the UE sending the SCGFailureInformationNR message can be modified.
2. Set the Rrc.t304 attribute to configure a waiting time for the completion of a successful NR Random Access procedure. When the timer for a successful NR Random Access procedure expires, the UE detects RLF and sends the SCGFailureInformationNR message to the Master Node.
3. Configure the DrbRlcUeCfg.dlMaxRetxThreshold attribute to define the number of the maximum allowed RLC retransmission in downlink. When this number is reached, the Secondary Node detects RLF and initiates the Secondary Node Release procedure.
4. Configure the DrbRlcUeCfg.ulMaxRetxThreshold attribute to define the number of the maximum allowed RLC retransmission in uplink. When this number is reached, the UE sends the SCGFailureInformationNR message to the Master Node which initiates the Secondary Node Release procedure.
5. Configure the DrbRlcUeCfg.tPollRetransmitDl attribute to set the time when a new poll request is sent by the UE. This occurs if no RLC status report is received from the Secondary Node. When the average ARQ RTT measured by the gNodeB is greater than the MO setting, the measured value is used and configured to the UE instead of the value in the t-PollRetransmit IE. This IE is only configurable at bearer setup. Configured bearers cannot be reconfigured.
6. Configure the DrbRlcUeCfg.tPollRetransmitUl attribute to set the time when a new poll request is sent by the Secondary Node. This occurs if no RLC status report is received from the UE. When the average ARQ RTT measured by the gNodeB is greater than the MO setting, the measured value is used and configured to the UE instead of the value in the t-PollRetransmit IE. This IE is only configurable at bearer setup. Configured bearers cannot be reconfigured.

## Appendix A.d: 19.Q4: Improved PDCP Flow Control Feedback and Buffer Management

This enhancement updates the PDCP flow control algorithm to use transmitted feedback

instead of delivered feedback.

| Feature Name:           | LTE-NR Dual Connectivity                |
|-------------------------|-----------------------------------------|
| Feature Identity:       | FAJ 121 4908                            |
| Value Package Name:     | NR Base Package                         |
| Value Package Identity: | FAJ 801 4002/5H R3  FAJ 801 4002/5LM R3 |
| Node Type:              | Baseband Radio Node                     |
| Access Type:            | NR                                      |

With this enhancement, PDCP flow control with LTE-NR downlink aggregation is performed

based on feedback from eNodeB and gNodeB lower layers. This is related to PDCP SN

submitted to the MAC layer for transmission, rather than on feedback related to PDCP SN

acknowledged as correctly received by the UE.

## Appendix A.bb: 24.Q4.0: Separation of EN-DC and NR-DC Configuration for Uplink Cell Group Switching

configurations for EN-DC and NR-DC.

| Access Type            | NR                         | NR                        |

Summary and Benefits

This feature enhancement allows for an optimized performance

for configuring

uplink

cell group

switching

in EN-DC and NR-DC deployments. The following parameters can be configured

separately for EN-DC and NR-DC deployments:

- Uplink cell group switching status
- Low-quality threshold for the uplink radio quality SINR measurement
- Low-quality hysteresis for the uplink radio quality SINR measurement

Capacity and Performance

The throughput can be tuned in EN-DC and NR-DC deployments independently by adjusting

the low-quality threshold of uplink cell group switching. This can be beneficial

when the cell capacity difference between the EN-DC MCG and the NR-DC MCG is

large.

## Appendix A.bd: Enhanced User Distribution for High Priority Frequencies in EN-DC

This feature enhancement introduces the mandatory higher-prioritized priority

option.

| Feature Name           | LTE-NR Dual Connectivity   |
|------------------------|----------------------------|
| Feature Identity       | FAJ 121 4908               |
| Value Package Name     | NR Base Package            |
| Value Package Identity | FAJ 801 4002               |
| Node Type              | Baseband Radio Node        |
| Access Type            | NR                         |
| Impact at Upgrade      | Mandatory                  |

Summary and Benefits

This feature enhancement allows the configuration of mandatory higher-prioritized

frequencies. The higher-prioritized frequencies depend on the preferred

configuration over the existing EN-DC user distribution model.

This feature enhancement has the following benefits:

- Improved efficiency of the use of NR frequency resources
- Possibility to always obtain the highest-priority frequencies based on the configuration of preferred frequencies

Capacity and Performance

This feature enhancement might affect the following PM counters:

- EUtranCellFDD.pmMeasConfigB1EndcDistribution
- EUtranCellTDD.pmMeasConfigB1EndcDistribution
- GUtranCellRelation.pmEndcSetupScgUeAtt
- GUtranCellRelation.pmEndcSetupUeAtt
- GUtranFreqRelation.pmEndcDistrFreqSel
- NRCellCU.pmEndcSetupScgUeAtt
- NRCellCU.pmEndcSetupUeAtt
- NRCellCU.pmRrcConnLevelMaxEnDc
- NRCellCU.pmRrcConnLevelSamp
- NRCellCU.pmRrcConnLevelSumEnDc
- NRCellCU.pmRrcConnLevelSumEnDc

This feature enhancement might affect the following EBS counters associated with the

NRCellCU MO instance:

- pmEbsEndcSetupDrbAtt
- pmEbsEndcSetupScgDrbAtt
- pmEbsEndcSetupScgUeAtt
- pmEbsEndcSetupUeAtt

For more information on EBS counter associated with the feature enhancement,

see Lists and Delta Lists.

Operation

This feature enhancement introduces the

EndcDistrProfile.mandatoryHiPrioGUtranFreqRef attribute to

define the list of higher-priority frequencies.

All frequencies defined in the

EndcDistrProfile.mandatoryHiPrioGUtranFreqRef attribute are

configured with a priority weight value. This feature enhancement can operate

separately or in combination with EN-DC user distribution attributes in the

following ways:

- Priority weight from the list of mandatory higher-priority frequencies only
- Priority weight from the mandatory higher-priority frequency list, and the mandatory frequency list
- Priority weight from the mandatory higher-priority frequency list, the mandatory frequency list, and the distribution model

As a result, these preferred frequencies are always prioritized over the

selected frequencies by the mandatory frequency list or the distribution model.

Reprioritization can be calculated using the enhanced distribution algorithm in the

following way:

- By defining the frequency in the mandatory higher prioritized frequency list
- By defining the frequency in the mandatory frequency list
- By defining the distribution model

The following cases can be configured:

- Reprioritized frequencies are only included in the mandatory higher priority frequency list.
- Reprioritized frequencies are included in the mandatory higher-priority frequency list and the distribution frequency list.
- Reprioritized frequencies are included in the mandatory higher-priority frequency list and the mandatory frequency list.
- Reprioritized frequencies are included in the mandatory higher-priority frequency list, the mandatory frequency list, and the distribution frequency list.

Interfaces

No impact.

Hardware

No specific hardware requirements.

Other Network Elements

Without the Release Independence Manager application, this feature enhancement

requires ENM version 25.2 or later. The Release Independence

Manager application also supports the feature enhancement on earlier supported ENM

versions.
